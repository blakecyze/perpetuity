What makes Hermes tick

A teardown of the only widely-used agent with a built-in learning loop — and a sober read on which parts are genuinely novel versus good plumbing wearing a halo.

Subject Nous Research · Hermes Agent v0.15.x Compiled 01 Jun 2026 For the perpetuity skill

§0 The one-sentence frame

Hermes is not a model and not a chatbot. It is an always-on runtime process — a Python program you install on a box (a $5 VPS, a homelab, a serverless backend) that wraps any LLM endpoint and runs continuously, reachable from a terminal or from Telegram/Discord/Slack. Everything people call "self-learning" hangs off the fact that it is a process with disk and a clock, not a single request/response turn.

That distinction is the whole story for us. Hold onto it — it is exactly the thing a Claude skill does not have, and Document 02 is about how far we can get without it.

The load-bearing fact Hermes learns because it is a long-lived process that can wake itself up while idle. Strip away the runtime and you are left with a set of protocols for what to write to disk and when — which, encouragingly, is portable.

§1 The closed learning loop

Nous describes a "closed loop" with four moving parts. They are independent mechanisms wired together, not one clever trick. Understanding them separately is what lets us cherry-pick.

create

Self-improvement loop

After a hard task, the agent writes a reusable skill describing how it solved it. Procedural memory, authored by the agent via the skill_manage tool.

maintain

The Curator

A background pass that ages, consolidates and archives agent-written skills so the library doesn't rot into near-duplicates.

remember

Persistent memory

Two tiny, bounded files (MEMORY.md, USER.md) injected into the system prompt at session start. Hard character caps force curation.

recall

Session search

Every past conversation in SQLite with full-text search. Cheap, instant, no LLM call — the agent greps its own history.

A fifth limb sits slightly outside: Honcho, an optional external "dialectic user-modelling" provider that builds a deepening model of who you are across sessions. It is one of eight pluggable memory backends, not core, and we treat it as out-of-scope for v1.

§2 Self-authored skills — the actual engine

This is the part everyone means by "it self-learns," and it is the most directly stealable. The agent has a tool, skill_manage, with the actions you'd expect: create, patch (preferred — token-cheap), edit, delete, plus file add/remove for multi-file skills. A skill is just a SKILL.md with YAML frontmatter, living under ~/.hermes/skills/, conforming to the agentskills.io open standard — the same standard Claude's own skills use.

When it decides to write one

The trigger conditions are explicit in the docs, and they are sensible heuristics rather than magic:

  • It completed a complex task (5+ tool calls) successfully.
  • It hit errors or dead ends and found the working path.
  • The user corrected its approach.
  • It discovered a non-trivial workflow worth not re-deriving.

Note what these have in common: each is a moment where effort was spent that would otherwise be lost at the end of the turn. That is the real definition of "worth remembering," and we lift it wholesale in Document 02.

Progressive disclosure keeps it cheap

Skills load in three tiers so a growing library doesn't bloat the context:

L0 · list (name+desc, ~3k tok)→ L1 · full SKILL.md on demand→ L2 · reference files only if needed

Only the lightweight index is ever "always on." This three-tier pattern is identical to how Claude's skills work, which is convenient — it means a perpetuity note library inherits the same token discipline for free.

§3 The Curator — and the detail that changes everything

Left alone, an agent that writes a skill after every hard task accumulates "dozens of narrow near-duplicates that pollute the catalog and waste tokens." The Curator exists to stop that. It is, in effect, a garbage collector for learned knowledge, and it is the single most important mechanism to copy — because the failure mode it prevents is the one most likely to sink perpetuity.

What it does

Two phases per run. First, deterministic ageing with no LLM involved: a skill unused for 30 days becomes stale; unused for 90 days it is moved to .archive/. Second, an LLM review — a single auxiliary-model pass (capped at 8 iterations, and you can pin it to a cheap model like a Flash) that surveys the agent-written skills and decides per-skill: keep, patch drift, consolidate overlapping ones, or archive.

It only ever touches agent-created skills — never bundled or hub-installed ones. It never deletes; the worst it does is archive into a recoverable folder. It takes a tar.gz snapshot before every real pass, supports one-command rollback, writes a human-readable REPORT.md per run, and lets you pin a skill to fence it off entirely.

The detail that reframes "background" The Curator is triggered by an inactivity check, not a cron daemon. On CLI session start, and on the gateway's recurring tick, Hermes asks: has interval_hours (default 7 days) passed and has the agent been idle for min_idle_hours (default 2h)? Only then does it fork. Even Hermes' "background learning" is really event-triggered work at quiet moments — not a thread humming away on its own.

This is enormously freeing for our design. The "passive background learning" Hermes advertises is, mechanically, "do maintenance at the next convenient boundary." A Claude skill cannot run a daemon — but it absolutely can act at boundaries (session start, task end, an explicit tick). We are closer to Hermes than the marketing implies.

§4 Memory: bounded on purpose

The instinct with memory is "store everything." Hermes does the opposite, and it's the more sophisticated choice. Two files, both with hard caps:

FileHoldsCap
MEMORY.mdAgent's own notes — environment facts, conventions, lessons learned~2,200 chars (~800 tok)
USER.mdUser profile — preferences, comms style, expectations~1,375 chars (~500 tok)

They're injected into the system prompt as a frozen snapshot at session start — captured once, never mutated mid-session, deliberately, to preserve the model's prefix cache. Writes hit disk immediately but only surface next session. The agent edits via a memory tool (add/replace/remove, substring matching, notably no read action — it just sees memory in its prompt).

When a file hits its cap, the agent must consolidate before adding — merge three "project uses X" notes into one dense entry. The cap isn't a limitation; it's a forcing function for curation. Compare this to session search, which is the opposite tier: unlimited, on SQLite FTS5, ~20ms, free (no LLM), used for "did we discuss X three weeks ago?" The split is the lesson:

Design principle to steal Keep a tiny, always-loaded index (bounded, curated, expensive-per-token so kept small) sitting on top of an unlimited, searched-on-demand archive (cheap, never in context until matched). Memory ≠ storage; memory is the bounded hot layer.

§5 What's actually novel — an honest read

Stripped of the "self-evolving" framing, every individual mechanism is mundane in the best way: bounded buffers, a deterministic state machine (active → stale → archived), event-triggered forks at idle, FTS over a SQLite table, recoverable archives, snapshots before mutation. None of it is a research breakthrough. A senior engineer would recognise all of it.

The novelty is the integration. Hermes is the first widely-adopted system to wire a complete capture → maintain → recall loop into an always-on agent runtime and ship it with the safety rails (never-delete, snapshot, pin, dry-run) that make self-modification tolerable in production. The achievement is product and discipline, not algorithm.

And it is genuinely double-edged — which is the part the launch posts skip:

  • Drift / entrenchment. A skill written from one fluky success becomes a confident instruction the agent re-applies. Self-authored knowledge can encode a wrong lesson as durably as a right one.
  • Proliferation. Capture-after-every-hard-task is a recipe for near-duplicate sprawl. The Curator isn't a nicety — it's load-bearing, and it's an LLM pass that costs tokens and can itself mis-consolidate.
  • Legibility. A library the agent rewrites is one where you have less visibility into why it now behaves a certain way. Kanso's manual, version-controlled approach is the opposite trade-off: more effort, full audit trail.
Carry-forward for perpetuity Copy the loop, but treat the maintenance pass and the safety rails as mandatory, not optional. The capture mechanism is the easy 20%; the curation, decay, provenance and rollback are the 80% that decide whether the thing helps or quietly poisons your workflow.
