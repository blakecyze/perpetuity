Turning it into a passive skill

A skill cannot run while you sleep — it has no process and no clock. This is the design for getting Hermes-grade continuity anyway, by acting at boundaries instead of running a daemon.

Artefact perpetuity · design v0.1 Sits beside kanso suite Standard agentskills.io / SKILL.md

§0 The constraint, stated plainly

Before any design, the honest part. A Claude skill is not a program. It is a folder of Markdown instructions that gets loaded into context during a conversation turn, when triggered. It has no thread, no event loop, no ability to wake itself at 3am and consolidate notes. Document 01 showed that Hermes "learns in the background" only because it is an always-on runtime with disk and a clock. A skill has neither.

So the literal goal — "a skill that learns and creates its own library in the background" — has a false premise. If we built it naively, it would do precisely nothing between your messages. That is the questionable bit of the brief, and it deserves a straight answer rather than an enthusiastic yes.

The reframe that makes it real Redefine "passive" as "zero extra effort from you," not "runs while you're away." A skill can be fully passive in the first sense: it can mandate that Claude recall from the library at the start of a task and capture to it at the end — automatically, without you ever asking. You do nothing; the learning still happens. It just happens at the edges of your turns, not in a vacuum.

And recall §3 of Document 01: even Hermes' Curator is triggered by an inactivity check at a boundary, not a true daemon. We are imitating a boundary-triggered system. The gap between "what Hermes really does" and "what a skill can do" is smaller than it first looks.

§1 The two-beat loop

Everything reduces to two behaviours the skill makes non-negotiable, plus a third that runs occasionally:

task starts→ RECALL: search library, load matches→ …work…→ task ends→ CAPTURE: write/patch a note if it earned one

— and, on an occasional boundary — CURATE: dedupe, age, consolidate —

Recall is what makes the library worth having; capture is what fills it; curate is what stops it rotting (the load-bearing lesson from Document 01). Miss any one and the system fails in a predictable way: no recall = a write-only diary nobody reads; no capture = an empty shell; no curate = near-duplicate sludge that poisons recall.

§3 Three tiers of "passive"

How autonomous it gets depends on how much infrastructure you're willing to attach. Ship them in order — each is useful alone, and each is a clean upgrade of the last.

Tier 1 · ship first

Pure skill — boundary behaviour only

works everywhere · incl. this chat UI · zero infra

The SKILL.md instructs Claude to search the notes directory before starting any non-trivial task, and to write/patch a note after finishing one that meets the capture criteria (§4). No daemon, no hooks — it "wakes" only when you're already talking to Claude, which is the honest limit. But within a session it is genuinely automatic and it persists across sessions through the files on disk.

SKILL.md trigger→ read INDEX.md→ grep matches→ work→ write note + update index

Limit: capture depends on Claude remembering to run the protocol. Skills can under-trigger. Mitigate with a "pushy" description and an explicit end-of-task checklist in the body.

Tier 2 · the sweet spot

Claude Code hooks — deterministic boundaries

Blake's daily driver · near-background · still legible

This is the tier that earns the word "passive" honestly, and it's the one I'd point you at. Claude Code fires 27+ lifecycle hook events (verified, v2.1.141+, May 2026) — and hooks fire on system events, not model whim, so execution is guaranteed in a way prompt instructions never are. The three that matter:

  • SessionStart — its stdout is injected as context Claude can see. Use it to auto-load the perpetuity index every session. That's recall, made deterministic.
  • Stop (Claude finished a turn) — fire an async: true hook that distils the just-finished transcript into a capture candidate out-of-band. Async hooks (shipped Jan 2026) run in the background without blocking. That is as close to "background learning" as the platform allows.
  • SessionEnd — run the consolidation/curate pass when you're done, the skill-world echo of Hermes' idle-triggered Curator.

Prior art exists: claude-mem already implements cross-session memory for Claude Code on exactly these five lifecycle hooks. Worth studying before building — it de-risks the whole tier.

Hook gotcha to design around A Stop hook that writes a file Claude then reacts to can create a loop. Keep the Stop hook passive — distil and write the note out-of-band (an agent/prompt handler or a headless call), don't feed it back into the live turn.
Tier 3 · only if you mean it

A real daemon — true always-on

honest answer to "runs while I sleep"

If genuine unattended, wake-itself-up learning is the actual requirement, stop trying to make a skill do it. Two clean routes:

  • Headless loop: a cron job on a $5 VPS running claude -p over your repos/notes, doing the consolidation pass on a schedule. This is Hermes' Curator pattern, rebuilt on Claude.
  • Fork Hermes itself. It's MIT-licensed and already model-agnostic — point it at an Anthropic endpoint and you inherit the entire loop (Curator, memory, FTS5, nudges) for free. Building a worse copy of an open-source thing you can just configure is the trap to avoid.
Proposed alternative — read this twice If "true background" is non-negotiable, forking/configuring Hermes onto an Anthropic endpoint is almost certainly the better use of your time than building perpetuity as a daemon. Build perpetuity as a skill only if you value portability, legibility, and tight integration with the kanso workflow over autonomy. Those are different products — pick deliberately.

§4 When to capture (and when not to)

Lifted directly from Hermes' triggers, because they're well-judged. Capture a note when:

  • A non-trivial workflow took 5+ meaningful steps and worked.
  • You recovered from a non-obvious error or dead end — the working path is the gold.
  • You corrected Claude's approach — corrections are the highest-signal events.
  • You hit a tool, config, or environment quirk worth not rediscovering.

Just as important, the skip-list — over-capture is the failure mode that creates sludge:

  • Trivial or one-step tasks Claude handles cold.
  • Facts that are cheaper to re-derive or web-search than to store.
  • Session ephemera — temp paths, one-off debugging state.
  • Secrets, keys, tokens — never. Scan on write, like Hermes does.

§5 Installing in parallel, without collision

The "blanket upgrade alongside any other library" goal is the easy part, because the ecosystem already standardised. perpetuity is a single skill folder conforming to agentskills.io / SKILL.md — the same standard kanso, Hermes, and Claude all speak. It drops into the skills directory next to everything else; no orchestration layer needed.

The thing that prevents collision is keeping its domain distinct. State it in the skill description so triggers don't fight:

SkillGovernsActs on
kanso suiteHow code is written, audited, committed — your taste, enforcedThe diff in front of you
perpetuityWhat is remembered across sessions — continuityThe notes library on disk

They compose rather than overlap: kanso shapes the work; perpetuity remembers how a hard piece of work got done so kanso doesn't re-solve it next month. Keep perpetuity's data in its own removable directory (~/.claude/perpetuity/ for global, or a repo-local .perpetuity/ for project-scoped memory) so the whole thing is additive and trivially uninstalled.

Provenance rule (steal from the Curator) Tag every perpetuity-authored note as agent-created and keep them in their own directory. The Curator's one sharp edge is that it can't tell a hand-authored skill from an agent-saved one. Don't inherit that bug — never let perpetuity's consolidation pass touch your kanso skills or anything you wrote by hand.

§6 The safety rails are the product

Document 01's verdict bears repeating as a build directive: capture is the easy 20%; the rails are the 80% that decide whether this helps or quietly harms. Non-negotiables for v1:

  • Never auto-delete. Worst case is archive to a recoverable folder. Mirror Hermes exactly.
  • Decay. Track last_used; age unused notes active → stale → archived on a deterministic clock before any LLM review touches them.
  • Confidence + supersedes. Every note carries a confidence and can point at a note it replaces — so a wrong lesson can be overridden, not silently re-applied. This is the antidote to entrenchment.
  • Consolidation is mandatory. Without a dedupe pass you get near-duplicate sprawl. It's not a v2 nicety.
  • Bounded index. Only the small INDEX.md is ever always-loaded; full notes load on match. Progressive disclosure, or context bloat eats the savings.
  • Pin + snapshot. Let yourself fence off a note; snapshot the library before any mutating pass.

§7 Recommended v1 scope

Don't build all three tiers at once. The pragmatic path:

  • v0.1 — Tier 1 pure skill: capture + recall at turn boundaries, manual /perpetuity-curate. Works in this chat and in Claude Code today. Prove the note schema (Document 03) and the recall actually helps before automating anything.
  • v0.2 — Add the Tier 2 Claude Code hooks: SessionStart recall injection, async Stop capture, SessionEnd curate. Now it's passive in the strong sense, on your daily driver.
  • v0.3+ — Only if you still want true autonomy, evaluate Tier 3 honestly against just running Hermes. Don't build a daemon out of pride.
Bottom line Yes, build perpetuity — but as a boundary-triggered continuity skill, not a background daemon. The skill form buys you portability, legibility, and seamless coexistence with kanso. If the daemon dream wins, configure Hermes instead. Either way, the maintenance and safety machinery — not the capture trick — is where the real work lives.
