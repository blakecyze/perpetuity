How to store the learned notes

You asked whether to use HTML instead of Markdown for the notes. The honest answer splits in two — and for the part that matters here, your hunch points the wrong way. Here's the evidence and the schema.

Question .md vs .html for the library Verdict Markdown notes · HTML review view Compiled 01 Jun 2026

§0 Two artefacts, not one

The confusion dissolves the moment you separate the two things in play, because they have opposite readers:

ArtefactWritten byRead byLives in
These three reportsClaude, onceYou — navigate, skim, actyour browser
The learned notesClaude, repeatedlyClaude — searched, loaded, patchedthe agent loop

The reports you're reading are HTML, and rightly so — that's the human-facing case where rich layout earns its cost. The learned notes are the exact opposite: a machine writes them, the same machine reads and rewrites them, a human rarely looks. That difference decides the format, and the current evidence is unusually clear about it.

§1 The case for Markdown (for the notes)

Everything that lives inside an agent loop pays a per-token tax on every read. HTML's tags, attributes, and wrappers are pure overhead the model must parse without semantic payoff. The 2026 benchmarks are consistent:

~3×
tokens, HTML vs clean MD, same content
20–40%
typical token cut converting HTML → MD
+35%
reported RAG retrieval accuracy on clean MD

On real-world pages the gap widens to roughly 68–87% fewer tokens. Beyond cost, structured Markdown measurably improves extraction and reduces hallucination versus verbose HTML — less markup noise means the model spends fewer cycles inferring structure and more synthesising. And the practical clinchers for a note library specifically:

  • Diff-friendly. Notes get rewritten and consolidated constantly. MD diffs are clean; HTML diffs are noise — closing tags and attribute churn obscuring the real change. For a git-tracked library this alone is decisive.
  • It's the native skill format. agentskills.io and Claude's own skills are SKILL.md. Notes in Markdown are skills — portable, hub-shareable, parsed best by the models trained on oceans of it.
  • Indexable. FTS and grep work cleanly over prose; chunking is predictable. Hermes stores sessions as plain text in SQLite for exactly this reason.

§2 The HTML counter-argument — and why it doesn't apply here

Your hunch isn't from nowhere. In May 2026 Thariq Shihipar (Claude Code team, Anthropic) published "HTML is the new Markdown," and it spread fast — Karpathy and Simon Willison both engaged. In one widely-repeated test, HTML beat Markdown in 17 of 20 head-to-head tasks. So why am I still saying Markdown?

Because of which 17. Every HTML win was a human-facing deliverable: status reports, code reviews, diagnostic summaries, data comparisons — things a person reads, navigates, and acts on, where collapsible sections, colour-coded status and tabbed layouts genuinely help. And the 3 cases Markdown held?

The exact carve-out that decides our question The three tasks Markdown won were the ones where "the output stays internal to an agent's loop and never reaches a human." That is the textbook description of a perpetuity note. Thariq's own operational rule: HTML when a third party reads but won't modify it; Markdown when the document is collaborative, indexed, or consumed by an automatic pipeline. Learned notes are all three of the latter.

So the debate doesn't contradict the recommendation — it confirms it. The HTML thesis is a recalibration for human-facing agent output, and the reports in this dossier obey it. It explicitly carves out the agent-loop case, which is where the notes live.

§3 Verdict

Markdown · source of truth The notes themselves, the INDEX, anything Claude reads, searches, patches, or commits to git. Non-negotiable.
HTML · on demand A generated browse / review surface for when you audit the library. Human-facing, so HTML earns its keep — but generated from the MD, never the store.

Store in Markdown. Don't store in HTML. The only place HTML belongs in perpetuity is the optional review view in §4 — and even that is derived, never the system of record.

§4 The hybrid — honouring the hunch

There is a real role for HTML here, just not as storage. A note library you can't see is one you won't trust or prune — and Document 02 made human review a safety rail. So: keep Markdown as the single source of truth, and let perpetuity generate an HTML dashboard on demand (/perpetuity-review) from the notes' frontmatter.

That review view is exactly Thariq's sweet spot — a human-facing deliverable — so it should be rich: cards per note, filter by tag/state, colour-coded confidence, "stale" and "superseded" badges, sorted by last-used. You get cheap, legible, diff-friendly storage and a genuinely nice audit surface, each format doing the job it's good at. This mirrors the living-HTML-dashboard pattern from the same Thariq talk, pointed at your own memory instead of a design system.

Rule of thumb Markdown is the database; HTML is a report you run against it. Never let the report become the database.

§5 The note schema

One Markdown file per note, SKILL.md-compatible so notes double as installable skills. YAML frontmatter carries the machine-readable metadata (the bit that powers recall, decay, and the HTML view); the body follows the Hermes When / Procedure / Pitfalls / Verification shape, which also happens to match your kanso house style.

# ~/.claude/perpetuity/notes/flutter-hot-reload-state-loss.md
---
name: flutter-hot-reload-state-loss
description: Fix for Riverpod state vanishing on hot reload — use
  when avatar cosmetic state resets mid-edit in the Avatone shop UI.
tags: [flutter, riverpod, avatone, hot-reload]
created: 2026-06-01
source_task: "debugging cosmetic preview resetting on save"
confidence: high          # high | medium | tentative
state: active             # active | stale | archived
use_count: 0
last_used: 2026-06-01
supersedes: null          # name of a note this replaces, if any
pinned: false
origin: agent-created     # never let curate touch hand-authored
---

## When to use
Riverpod providers holding ephemeral UI state reset on hot reload,
losing the in-progress cosmetic selection.

## Procedure
1. Move the selection into an autoDispose provider keyed by item id.
2. Persist the draft to the cosmetic_draft box before rebuild.
3. Rehydrate in build() from the box, not from a field.

## Pitfalls
- A plain StateProvider survives reload but leaks across items.
- Don't key by index — reorderable grid breaks it.

## Verification
Edit a cosmetic, save, trigger hot reload — selection persists.

The frontmatter is the entire engine: description drives recall matching, tags drive search, state/last_used drive decay, confidence/supersedes are the anti-entrenchment rails, origin protects your hand-written work from the consolidation pass.

§6 The index — the always-loaded layer

Mirroring Hermes' bounded MEMORY.md and the L0 skills list (~3k tokens), only a thin index is ever loaded by default. Full notes load on match. Generate it from frontmatter so it never drifts:

# ~/.claude/perpetuity/INDEX.md  (generated — do not hand-edit)
- name: flutter-hot-reload-state-loss
  desc: Riverpod state vanishing on hot reload (Avatone shop)
  tags: [flutter, riverpod, avatone]  state: active  conf: high
- name: cra-gst-payment-matching
  desc: CRA payment not matching a GST34 return
  tags: [tax, cra, sole-prop]        state: active  conf: medium

Keep the index hard-capped like Hermes caps memory — a forcing function for consolidation. When it grows past budget, that's the signal for a curate pass, not a bigger budget. The use_count/last_used counters double as the .usage.json-style telemetry that the decay state machine and the HTML review view both read.

One-line answer to the brief Markdown for every note and index (agent-loop, indexed, diff-tracked, skill-compatible); HTML only for the on-demand human review dashboard. Your instinct to use HTML was right — but for the reports, not the memory.
